We need to improve the Purchase Order PDF/export in ShelfSense so it becomes more professional and SaaS-ready, not only suitable for Salted Grills.

Current situation:
- PO is generated automatically from low-stock items.
- It suggests ordered quantities based on restock level.
- Current PDF includes PO number, date, branch, status, supplier, line items, ordered/received/remaining, unit cost, total value.
- Issue: unit cost and total value show Rs 0 when cost is missing.
- PDF is functional, but it needs to look more like a professional procurement document suitable for any business using ShelfSense.

Please implement the following improvements carefully without breaking existing PO creation, receiving, or stock logic.

1. Improve PO PDF header

Add a professional header section with:

- Business / workspace name
- Branch name
- PO number
- PO date
- PO status
- Supplier name
- Expected delivery date, if available
- Created by, if available
- Approved by, if available

If any field is not available, show “—” instead of leaving awkward blank space.

The header should look clean and print-friendly.

2. Add PO summary box near the top

Add a summary section before line items:

- Total unique items
- Total ordered quantity
- Total received quantity
- Total remaining quantity
- Estimated PO value
- Total received value

If pricing is missing, estimated value should show “—” or “Pricing not set” instead of Rs 0.

3. Fix unit cost and total value display

Do not show Rs 0 by default when item cost is missing.

Logic should be:

- If unitCost exists and is greater than 0, show formatted value like “Rs 1,250”
- If unitCost is missing/null/0, show “—” or “Not set”
- If total cannot be calculated because unit cost is missing, show “—”
- Only show Rs 0 if the actual saved value is intentionally 0 and the system can clearly distinguish that.

This is important because Rs 0 makes the PO look incomplete or incorrect.

4. Add stock intelligence columns

The PO should explain why items are being ordered.

Add these columns if data is available:

- Current Stock
- Restock Level
- Suggested Qty
- Ordered Qty
- Received Qty
- Remaining Qty
- Unit Cost
- Total Value

If the PDF becomes too wide, use a clean compact layout. Priority columns should be:

Item, Current Stock, Restock Level, Suggested Qty, Ordered Qty, Unit Cost, Total Value

Received and Remaining can be included if the PO is in receiving/partial/closed state.

5. Separate suggested quantity from final ordered quantity

The system-generated suggested quantity and the user-finalized ordered quantity should be separate.

Example:
- Suggested Qty: calculated from restock logic
- Ordered Qty: actual quantity saved on PO

If both are currently the same, still display both if data exists. This is important for SaaS because users may adjust the system suggestion before confirming PO.

6. Add item category grouping

If item category exists, group PO line items by category.

Example:

Cleaning Supplies
- Acid for Toilet
- Air Freshener
- Dishwashing Liquid

Dairy
- Milk Carton
- Milk Pack Cream

Sauces & Condiments
- Chilli Garlic
- HP Sauce
- Ketchup

If category does not exist, show under “Uncategorized”.

This will make long POs easier to read.

7. Add supplier-facing details

If supplier profile has these fields, show them in the supplier section:

- Supplier name
- Contact person
- Phone
- Email
- Address

If these fields do not exist yet, do not break the PDF. Just show the available fields and use “—” for missing values.

8. Add buyer/business details

Add buyer details from workspace/branch settings if available:

- Business name
- Branch name
- Address
- Phone
- Email

This makes the PO suitable for sending to suppliers.

9. Add approval section

At the bottom of the PDF, add a clean approval/signature area:

Prepared By: __________
Reviewed By: __________
Approved By: __________

Also include:
Approval Status: Draft / Pending Approval / Approved / Sent / Partially Received / Closed / Cancelled

Use existing PO status where available. Do not invent a new status system unless needed. If approval workflow does not exist yet, only make the PDF ready for it visually.

10. Add receiving section

At the bottom, add receiving fields:

Received By: __________
Receiving Date: __________
Supplier Invoice/Bill No: __________
Remarks: __________

This should help users use the printed PO during actual stock receiving.

11. Improve page headers and footers

For multi-page PDF, each page should have a compact repeating header:

PO Number | Supplier | Branch | Status | Page X of Y

Footer should include:

Generated by ShelfSense
Generated date/time

Avoid showing browser artifacts like “about:blank” or browser print timestamp if possible.

12. Improve visual design

Make the PDF clean, modern, and professional:

- Better spacing
- Clear section headings
- Light borders
- Proper table alignment
- Amounts right-aligned
- Quantities center/right-aligned
- Item names left-aligned
- Use consistent font size
- Avoid too much empty space
- Make it print-friendly on A4

13. Add SaaS flexibility

This PO should not be hardcoded for Salted Grills.

Use dynamic workspace/business settings wherever possible:

- Currency symbol/code
- Business name
- Branch name
- Supplier details
- Date format
- Timezone if already configured

If currency is missing, default to Rs for now but keep code flexible.

14. Backward compatibility

Very important:
- Do not break existing PO creation.
- Do not break existing receiving logic.
- Do not break existing stock deduction/addition logic.
- Do not break current database records.
- If new fields are needed, add them safely using migration with nullable/default fields.
- Existing POs should still open and export even if new data fields are missing.

15. Testing required

After implementation, test:

- Draft PO with missing prices
- Draft PO with prices set
- PO with many line items across multiple pages
- PO with categories
- PO without categories
- PO with supplier details
- PO with minimal supplier details
- PO receiving/partial receiving state, if supported
- PDF print preview on A4
- Mobile/tablet view if PDF preview exists in browser

Expected outcome:
The PO PDF should look like a professional SaaS-grade procurement document. It should clearly show why items are being ordered, what the system suggested, what the user ordered, estimated cost where available, supplier/business details, approval area, and receiving area.

Please first inspect the existing PO model, PO PDF/export component, supplier model, inventory item model, workspace/branch settings, and current restock quantity logic before making changes. Then implement the improvement in the safest and cleanest way.
