We need to build a new premium feature in ShelfSense called:

“Smart Receiving Match Engine”

This feature should enhance the existing Purchase Order receiving workflow with invoice OCR, AI-assisted item matching, tax normalization, and GRN creation.

====================================================
BACKGROUND / CURRENT FLOW
=========================

ShelfSense currently has a Purchase Order receiving workflow.

Current process:

1. User creates a Purchase Order.
2. User clicks “Receive Items”.
3. User goes to the Receive Stock page.
4. User selects a PO.
5. PO items populate automatically.
6. Receiving quantity is auto-filled based on PO quantity but can be changed by the user.
7. User manually enters:

   * Unit cost
   * Tax
   * Batch number
   * Expiry date
8. User confirms receiving.
9. Inventory is updated.

This flow is correct and should remain the core procurement receiving flow.

====================================================
IMPORTANT ARCHITECTURE RULE
===========================

Procurement inventory must ONLY increase through:

Purchase Order Receiving / GRN

OCR must NOT update stock directly.

OCR should only prepare a draft receiving document by extracting invoice data and auto-filling fields.

Final inventory update must happen only after the user reviews and confirms receiving / creates GRN.

Do not create a separate OCR Stock-In workflow.

This feature must enhance the existing PO Receive Stock page.

====================================================
FEATURE GOAL
============

When supplier goods arrive with an invoice, the user should be able to upload the supplier invoice on the Receive Stock page.

The system should:

1. Extract invoice data using OCR.
2. Compare invoice items with PO items and inventory items.
3. Suggest the best item matches with confidence percentage.
4. Auto-fill receiving quantity, cost, tax, batch, and expiry where possible.
5. Calculate unit cost including tax correctly.
6. Show quantity/cost/tax variances.
7. Let the user review and edit everything.
8. Create GRN only after confirmation.
9. Attach the invoice to the GRN for audit trail.

====================================================

1. User opens Receive Stock page.

====================================================
INVOICE TAX AND COST NORMALIZATION
==================================

This is very important.

Supplier invoices may show tax in different formats.

The system must normalize all invoice cost data into these fields:

* unitCostExclTax
* unitTax
* unitCostInclTax
* lineSubtotalExclTax
* lineTaxTotal
* lineTotalInclTax
* taxRate
* taxMode

Supported invoice formats:

1. Tax per unit
2. Tax per line item
3. Tax only at invoice total level
4. Price already includes tax
5. No tax

====================================================
LINE-LEVEL TAX FORMAT
=====================

Some supplier invoices have columns like:

Item Name | Qty | Unit Price | Total Price | Tax | Sum

In this format:

* Unit Price is usually excluding tax
* Total Price is Qty × Unit Price excluding tax
* Tax is the collective tax amount for the full line quantity
* Sum is Total Price + Tax

Example:
Qty = 10
Unit Price Excl. Tax = 100
Total Price Excl. Tax = 1,000
Line Tax Total = 180
Line Total Incl. Tax = 1,180

The system must calculate:

unitTax = lineTaxTotal / quantity

unitCostInclTax = lineTotalInclTax / quantity

Alternative:
unitCostInclTax = unitCostExclTax + unitTax

Example calculation:
Unit Tax = 180 / 10 = 18
Unit Cost Incl. Tax = 1,180 / 10 = 118

The receiving table should show:

* Qty = 10
* Unit Cost Excl. Tax = 100
* Line Subtotal Excl. Tax = 1,000
* Line Tax Total = 180
* Unit Tax = 18
* Unit Cost Incl. Tax = 118
* Line Total Incl. Tax = 1,180

====================================================
TAX ONLY AT INVOICE TOTAL LEVEL
===============================

If tax is only given at invoice total level and not per line:

Example:
Line 1 subtotal = 1,000
Line 2 subtotal = 2,000
Invoice subtotal = 3,000
Invoice tax = 540
Invoice total = 3,540

Then allocate tax proportionally:

Line 1 tax = 540 × 1,000 / 3,000 = 180
Line 2 tax = 540 × 2,000 / 3,000 = 360

Then calculate unitTax and unitCostInclTax for each line.

Mark such tax as:
taxMode = “allocated_from_invoice_total”

Show a warning:
“Tax was allocated proportionally from invoice-level tax. Please review before confirming.”

====================================================
TAX-INCLUSIVE PRICE FORMAT
==========================

If the invoice price already includes tax:

Store extracted unit price as:
unitCostInclTax

If tax rate is known, back-calculate:

unitCostExclTax = unitCostInclTax / (1 + taxRate)

unitTax = unitCostInclTax - unitCostExclTax

Example:
Unit Cost Incl. Tax = 118
Tax Rate = 18%

unitCostExclTax = 118 / 1.18 = 100
unitTax = 18

Mark:
taxMode = “tax_inclusive_price”

====================================================
NO TAX FORMAT
=============

If no tax is found:

unitTax = 0
lineTaxTotal = 0
unitCostInclTax = unitCostExclTax
lineTotalInclTax = lineSubtotalExclTax
taxMode = “no_tax”

====================================================
INVENTORY COST BASIS SETTING
============================

Add a workspace-level setting:

Inventory Cost Basis:

1. Including Tax
2. Excluding Tax

Default should be:
Including Tax

Reason:
For restaurant inventory use cases, the owner usually needs to know actual purchase cost paid per unit.

If Inventory Cost Basis = Including Tax:

* Inventory valuation should use unitCostInclTax

If Inventory Cost Basis = Excluding Tax:

* Inventory valuation should use unitCostExclTax

However, both values should always be stored for reporting and audit.

====================================================
MATCHING ENGINE REQUIREMENTS
============================

The system should match extracted invoice lines with PO and inventory items.

Compare invoice line against:

* PO item name
* Inventory item name
* SKU
* Barcode
* Supplier item code
* Supplier item alias
* Brand
* Purchase unit
* Pack size
* Previous invoice history
* Supplier-item mapping

Confidence score should consider:

* Exact SKU/barcode match
* Supplier item alias match
* Name similarity
* Brand similarity
* Unit similarity
* Pack size similarity
* Supplier match
* PO presence
* Previous confirmed mapping
* Historical cost range

Confidence behavior:

* 90% to 100% = High confidence, auto-suggest
* 70% to 89% = Medium confidence, suggest but mark as Needs Review
* Below 70% = Low confidence, do not auto-map unless user confirms
* No match = show as Extra Invoice Item

Examples:
Invoice item: “National Chilli Garlic 4KG”
Inventory item: “Chilli Garlic (4 KG) - National / bottle”
Expected confidence: High

Invoice item: “Multipurpose Cleaner”
Inventory item: “All Purpose Cleaner”
Expected confidence: Medium, needs review

BATCH AND EXPIRY TRACKING
=========================

OCR should extract batch number and expiry date where available.

But many invoices will not include batch/expiry.

Inventory item master should have setting:

Validation:

* If item requires expiry tracking and expiry date is missing, user must enter expiry date manually before confirming.
* If item does not require expiry tracking, expiry field should be optional.
* Batch number should be optional unless the item or workspace setting requires it.

Examples:
Expiry required:

* Chicken
* Dairy
* Frozen items
* Sauces
* Perishable food items

Expiry optional:

* Cleaning items
* Packaging items
* Non-consumables
