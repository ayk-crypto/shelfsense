Add a SaaS-ready Physical Inventory Count Frequency and Reminder feature.

Context:
The application is a multi-tenant inventory management SaaS. The feature must be designed generically for any business using the platform, not for one specific restaurant or company.

Goal:
Allow each tenant/workspace/business to configure how often physical inventory counts should be performed. The system should calculate reminders based on the last completed physical count, not fixed calendar dates.

Important Business Logic:
The reminder schedule must follow a rolling cycle based on the last completed physical count.

Example:
If a tenant selects Monthly frequency and completes physical inventory on 14 May, the next reminder should be due around 14 June.
It should NOT remind on 1 June.

This is correct because the business selected a one-month frequency, so the system should maintain a one-month gap from the actual completion date.

3. Next Due Date Calculation

Create a backend helper function:

calculateNextPhysicalCountDueDate(lastCompletedAt, settings)

Rules:
- If reminders are disabled, return null.
- If no previous completed count exists:
  - calculate first due date from settings.created_at + configured frequency.
- If a previous completed count exists:
  - Weekly = lastCompletedAt + 7 days
  - Bi-weekly = lastCompletedAt + 14 days
  - Monthly = lastCompletedAt + 1 calendar month
  - Quarterly = lastCompletedAt + 3 calendar months
  - Custom days = lastCompletedAt + custom number of days
  - Custom weeks = lastCompletedAt + custom number of weeks
  - Custom months = lastCompletedAt + custom number of calendar months

Important:
Do not calculate monthly reminders from the 1st day of the next month.
Do not use fixed calendar resets by default.
Always calculate the next due date from the actual completed count date.

Examples:
- Completed: 2026-05-14
  Frequency: Monthly
  Next due: 2026-06-14

- Completed: 2026-01-31
  Frequency: Monthly
  Next due: 2026-02-28 or the last valid day of the next month.

- Completed: 2026-05-14
  Frequency: Weekly
  Next due: 2026-05-21

4. Reminder Trigger Logic

Add a backend scheduled job or daily reminder check using the existing app architecture.

The job should run daily and check:
- reminders are enabled
- next_due_at is not null
- current date >= next_due_at - reminder_lead_days
- no active physical count already exists for the same tenant/workspace/location
- reminder for the same due cycle has not already been sent, if reminder deduplication exists

If conditions match:
- create an in-app notification
- show dashboard alert
- optionally trigger email notification if email notification system exists
- optionally trigger push notification if supported later

Notification examples:
- “Physical inventory count is due. Please start a new count to verify stock accuracy.”
- “Physical inventory count is due in 3 days.”
- “Physical inventory count is overdue since [date]. Please complete stock verification.”

6. Physical Count Workflow

Create or improve the physical count workflow so it is operationally smooth for any tenant.

Suggested flow:

Inventory → Physical Counts → Start Count

Step 1: Select location/branch, if applicable.

Step 2: Select count scope:
- Full inventory
- Category-wise count
- Selected items
- Low-stock items
- High-value items
- Expiry-sensitive items

Step 3: Generate count sheet.

Step 4: User enters counted quantity.

Step 5: System compares:
- System quantity
- Counted quantity
- Difference
- Variance value

Step 6: User adds variance reason:
- Wastage not recorded
- Internal usage not recorded
- Loss/damage
- Receiving mismatch
- Sales/POS consumption mismatch
- Counting mistake
- Unit conversion issue
- Other

Step 7: Manager/admin reviews variances.

Step 8: Approved variances create controlled inventory adjustment entries.

Step 9: On final completion:
- save completed_at
- update last_completed_at in physical inventory settings
- recalculate next_due_at using completed_at + selected frequency
- create audit log
- update dashboard reminder state

7. Inventory Safety Rule

Do not blindly overwrite inventory when a physical count is completed.

Use a controlled adjustment flow:
- Count is completed
- Variances are generated
- Variances are reviewed
- Approved variances create inventory adjustment records
- Inventory is updated through adjustment entries
- Audit trail is maintained

This prevents accidental stock corruption and supports SaaS-grade accountability.

10. Multi-Tenant SaaS Requirements

This feature must be fully tenant-isolated.

Requirements:
- No tenant can view or modify another tenant’s physical count settings.
- Physical count records must be scoped by tenant/workspace.
- Notifications must only go to users belonging to the correct tenant/workspace.
- Location-specific counts must only show relevant items.
- API routes must validate tenant/workspace access.
- Background jobs must process tenants safely without cross-tenant leakage.
- All database queries must include tenant/workspace scoping.

12. Acceptance Criteria

- Tenant admin can enable/disable physical inventory reminders.
- Tenant admin can set frequency as weekly, bi-weekly, monthly, quarterly, or custom.
- System calculates next_due_at from the last completed physical count date.
- If monthly frequency is selected and count is completed on 14 May, next due date becomes 14 June.
- Reminder does not reset to the 1st of the next month.
- Dashboard shows physical count reminder status clearly.
- System creates notifications when count is upcoming, due, or overdue.
- Starting a count prevents duplicate active counts for the same tenant/location.
- Completing a count updates last_completed_at and recalculates next_due_at.
- Physical count variances are reviewed before inventory is adjusted.
- Inventory is updated only through approved adjustment entries.
- All important actions are captured in audit logs.
- Feature works safely across multiple tenants/workspaces.
- UI is responsive and suitable for mobile counting operations.

Please implement this feature using the existing SaaS architecture, tenant/workspace isolation, authentication, authorization, API conventions, database migration style, UI components, notification system, audit logging approach, and existing inventory adjustment patterns.
