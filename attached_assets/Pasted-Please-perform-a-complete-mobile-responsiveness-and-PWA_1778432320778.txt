Please perform a complete mobile responsiveness and PWA readiness review of ShelfSense.

The application is currently partially responsive, but some screens still require horizontal scrolling on mobile. This creates poor UX, especially for restaurant/store users who will use ShelfSense from mobile devices.

The main issue observed is on the Reports section, where not all reports, tables, filters, cards, and report outputs are properly visible on mobile screens. However, please do not only fix the Reports page. Perform a complete smoke test across the whole application.

Objective

Make ShelfSense fully mobile responsive, practical, and PWA-ready so users can comfortably use it from mobile browsers and installed home-screen app mode.

The application should work properly on:

360px width mobile screens
375px width iPhone-style screens
390px–430px modern mobile screens
Tablets
Desktop screens

Reports page requirements

The Reports section must be redesigned for mobile usability.

On desktop, reports can remain table-based.

On mobile, reports should use a mobile-friendly layout such as:

Stacked cards instead of wide tables
Collapsible report sections
Horizontally scrollable table only inside a contained table wrapper, not causing full-page horizontal scroll
Sticky/visible filters where useful
Compact summary cards
Clear labels and values
Proper spacing between report rows

For example, instead of showing a wide table like:

Item | Category | Opening Stock | Received | Issued | Current Stock | Value | Location

On mobile, show it as a card:

Item Name
Category: Grocery
Opening: 20 packet
Received: 10 packet
Issued: 5 packet
Current: 25 packet
Value: Rs 2,500
Location: Main Branch

Mobile UX expectations

Please make mobile UX feel intentional, not just squeezed.

Requirements:

No full-page horizontal scrolling.
All pages should fit within mobile viewport.
Buttons should be easy to tap.
Forms should stack vertically on mobile.
Tables should become cards or be contained within safe scroll wrappers.

Text should not overlap or cut off.
Modals should fit within screen height and allow vertical scroll.
Charts should resize properly.
Important actions should remain visible and usable.
Pages should look clean on both mobile and desktop.

Technical instructions

Please inspect the existing frontend layout system before making changes.

Look for common causes of mobile overflow, such as:

width: 100vw;
min-width: 900px;
fixed table widths;
large grid columns without breakpoints;
nowrap text;
overflow-visible containers;
buttons in one row without wrapping;
cards using fixed widths;
charts without responsive containers;

Please replace these with responsive patterns:

max-width: 100%;
overflow-x: hidden only where safe;
grid-template-columns with responsive breakpoints;
flex-wrap;
min-width: 0;
truncate long text properly;
responsive table/card pattern;
contained overflow-x-auto for tables;

For React/Tailwind components, please ensure:

Use w-full
Use min-w-0
Use responsive grids like grid-cols-1 md:grid-cols-2 lg:grid-cols-4
Use flex-col sm:flex-row
Use overflow-x-auto only inside table wrappers
Avoid fixed pixel widths unless necessary
Use responsive padding such as p-3 sm:p-4 lg:p-6
Use smaller text and spacing where appropriate on mobile
Use card-based layout for dense data on mobile

Acceptance criteria

Implementation will be considered complete when:

No major app screen has full-page horizontal scroll on mobile.
Reports page is fully usable on mobile.
Tables are either converted into mobile cards or safely contained.
Filters and buttons stack properly on mobile.
Sidebar/navigation works cleanly on mobile.
Forms and modals are usable on small screens.
PWA manifest and installability basics are verified.
App looks professional on mobile, tablet, and desktop.
A mobile smoke test checklist is documented.
Existing desktop layout should not be broken.

Please start by auditing the layout and identifying all overflow sources, especially in Reports, Purchases, Reorder Suggestions, Stock Activity, and Admin pages. Then implement reusable responsive components so the same problem does not appear again on future pages.
